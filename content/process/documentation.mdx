## ðŸ“ File Naming Rules

**Format:** `YYYYMMDD_type_shortdesc.ext`

### Examples

âœ… **Good:**
- `20251125_photo_market_entrance.jpg`
- `20251203_interview_vendor_amina.mp3`
- `20251209_prototype_collection_drum.jpg`
- `20251210_video_demo.mp4`

âŒ **Bad:**
- `IMG_1234.jpg` (no context)
- `photo.jpg` (no date or description)
- `Market Photo (1).jpg` (spaces and special characters)
- `vendor interview.mp3` (spaces, no date)

### File Type Codes

- `photo_` â€” Photographs
- `video_` â€” Video recordings
- `audio_` â€” Audio recordings/interviews
- `screenshot_` â€” App/website screenshots
- `diagram_` â€” System maps, sketches
- `consent_` â€” Consent forms

## ðŸ”„ PR & Review Flow

### Step-by-Step Process

1. **Team uploads** to Google Form with all required files

2. **Kamal (Hub Git operator) receives** notification
   - Creates a new branch: `team-[teamname]`
   - Copies files to appropriate folders
   - Opens Pull Request (PR)

3. **Reviewer checks** submission against checklist
   - All required files present?
   - Files named correctly?
   - Evidence meets minimum requirements?
   - Consent documented?

4. **If changes needed:**
   - Reviewer comments on PR
   - Team resubmits via form
   - Kamal updates the branch

5. **If approved:**
   - Hub Lead or organizer merges final PR
   - Team's work is published to main site
   - Team receives confirmation

### Timeline

- **Initial review:** Within 24 hours of submission
- **Revision requests:** Team has 48 hours to resubmit
- **Final merge:** Within 24 hours of approval

## ðŸŽ¯ Quality Standards

### Minimum Acceptable Evidence

**Photos:**
- Clear, well-lit, in focus
- Show real context (not staged)
- Include people interacting with solution (with consent)

**Quotes:**
- From real users/stakeholders (not team members)
- Include name or "Anonymous" + role
- Specific feedback, not generic praise

**Reflection:**
- Honest about what worked AND what didn't
- Specific learnings, not vague statements
- Clear next steps or recommendations
